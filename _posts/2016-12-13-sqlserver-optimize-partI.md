### 第一步：在数据库中建立适当的索引

有些人可能会认为实现适当的索引是否应该是数据库性能优化过程的第一步，但我想首先在数据库中正确应用索引，因为以下两个原因：

>* 这将允许您在生产系统中以最快的时间实现最佳的性能
>* 在数据库中应用/创建索引不需要你进行任何应用程序修改，因此不需要任何构建和部署

### 什么是索引

我相信你知道什么是索引。但是，我看到很多人对此不清楚。所以，让我们再次尝试理解索引。让我们读一个小故事

很久以前，古城有一个大图书馆。它有成千上万的书，但书的货物没有按任何顺序在书架上排列。每当一个人向图书馆员要求一本书时，图书馆员都没有办法，只能检查每本书，找到他想要的书。图书管理员需要花好几个小时找到这本书，借书人不得不等待很长时间。

[嗯...听起来像没有主键的表。当在表中搜索数据时，数据库引擎必须扫描整个表以找到对应的行，其执行非常慢。]

图书管理员生活得越来越凄惨，因为书的数量和人要求书的数量日益增加。然后有一天，一个聪明的家伙来到图书馆，看到图书馆员的可衡量的生活，他建议为每本书编号，并根据他们的编号安排书架，图书管理员说:"我能得到什么好处"，聪明的家伙回答："好吧，现在如果有人给你一个书号，并要求那本书，你将能够快速找到包含书的编号的书架，在那个架子里，你可以很快找到那本书，因为书根据他们的书号排列"

[对书籍编号听起来像在数据库表中创建主键，在表中创建主键时，创建聚集索引树，包含表行的所有数据页根据它们的主键值在文件系统中进行排序，每个数据页包含行，它们也根据它们的主键值在数据页内排序，每次从表中请求任何行时，数据库服务器首先使用聚集索引树查找相应的数据页（就像先找书架）然后在包含主键值的数据页中找到所需的行（就像在书架里找到这本书）]

"这正是我需要的！" 激动的图书馆员立即开始编号的书，并将它们安排在不同的书架上，他花了一整天来做这个安排，但在一天结束的时候，他测试并发现根据数字找书几乎没花多少时间，图书管理员非常高兴。

[这正是在表中创建主键时发生的情况，在内部，创建聚集索引树，并且根据主键值在数据文件内对数据页进行排序，您可以轻松地理解，只能为表创建一个聚簇索引，因为数据可以仅使用一个列值作为标准进行物理排列（主键）。这就像书只能使用一种标准(这里是书号)]

等等！问题还没有完全解决，第二天，一个人用书名找一本书(他不知道书的编号，他只知道书名)。
可怜的图书管理员扫描了所有的编号都没有办法，他在第67层的书架上找到了这本书，图书管理员花了20分钟找到这本书，当没有安排在货架上，他花了2-3个小时找一本书，所以这是一个改进，但是，与使用编号（30秒）搜索图书的时间相比，这20分钟看起来非常慢。所以他问这个聪明的家伙怎么改进。

[这就像有一个产品表，其中有一个主键 ProductID，但是在表中没有其他索引，当使用产品名称搜索产品时，数据库引擎不能根据扫描文件中所有物理排序的数据页来找到所需的项目]

这个聪明的家伙告诉图书管理员:"因为你已经使用他们的序列号排列你的书，你不能重新排列，更好的办法是创建目录或索引，这样就能获得所有书名以及其相应的序列号，在此目录中，按字母数字排列书名，并使用每个字母表对书名进行分组，以便任何人想要找到名为“数据库管理系统”的书，您只需按照以下步骤找到书"：

>* 跳转到您的书名“目录”的部分“D”，并在那里找到书名。
>* 阅读相应的书籍序列号，并使用序列号找到书（你已经知道如何做到这一点）。

“你是个天才！” ，图书馆大声喊道。花了几个小时，他立即创造了“书名”目录，并用快速测试，他发现他只需要一分钟（在“图书名称”目录中查找图书的序列号30秒，使用序列号查找图书30秒）根据书名找到书。

图书馆员认为，人们可能会使用诸如图书名称和/或作者姓名等其他标准来查找图书，所以他为作者名称创建了类似的目录，并创建了这些目录，图书管理员可以在一分钟内使用通用标准（序列号，书名，作者姓名）找到任何书。图书馆员的苦难很快就结束了，许多人开始在图书馆收集书籍，因为他们可以很快得到的书，图书馆变得非常受欢迎。

图书管理员开始过着幸福的生活。故事结束了。

到这个时候，我相信你真的已经明白了什么是索引，为什么他们很重要，以及他们的内部工作是什么，例如，如果我们又一个 "Products" 表，以及创建聚簇索引（在表中创建主键时自动创建），我们应该在ProductName列上创建一个非聚集索引，如果我们这样做，数据库引擎将为非聚集索引（如故事中的“book name”目录）创建索引树，其中产品名称将在索引页中排序，每个索引页将包含一系列产品名称及其对应的主键值，当在搜索条件中使用产品名称搜索产品时，数据库引擎将首先搜索产品名称的非聚集索引树以查找图书的主键值，一旦找到，数据库引擎然后用主键搜索聚集索引树以找到正在搜索的实际项目的行。

下面是索引书是如何工作的

<img src="https://www.codeproject.com/KB/database/OptimizeDBUseIndexing/IndexTree.JPG"  style="width:750px;"/>

这被称为B +树(Balanced tree).中间节点包含一系列值，并指示SQL引擎在树中搜索特定索引值，从根节点开始，叶节点是包含实际索引值的节点，如果这是一个聚集索引树，叶节点是物理数据页，如果这是非聚集索引树，叶节点包含索引值以及聚集索引键（数据库引擎用来在聚集索引树中查找相应的行）。

在索引树中找到期望值并从那里跳到实际行对于数据库引擎花费极少量的时间，所以，索引通常改进了数据检索操作。

请按照以下步骤操作，以确保数据库中的索引正确：

### 确保数据库中的每个表都有一个主键

确保每个表都创建了一个聚集索引(因此，根据主键字段在磁盘中物理地排序表的相应页).使用主键从表执行的任何数据检索操作或对主键字段或where子句中指定的任何主键值范围的任何排序操作都非常快速。

### 在列上创建非聚簇索引

>* 经常在搜索条件中使用
>* 用于连接其他表
>* 用作外键字段
>* 具有高选择性（从特定值的总行数返回低百分比（0-5％）行的列）
>* 用于ORDER BY子句
>* XML类型（需要创建主索引和辅助索引;更多信息请参见后续文章）

以下是表上的索引创建命令的示例：

```sql
CREATE INDEX
NCLIX_OrderDetails_ProductID ON
dbo.OrderDetails(ProductID) 
```

或者，您可以使用SQL Server Management Studio在所需的表上创建索引

### 第二步：创建适当的覆盖索引

你已经在数据库中创建了所有适当的索引，假设，在此过程中，您已在Sales（SelesID，SalesDate，SalesPersonID，ProductID，Qty）表中的外键列（ProductID）上创建索引，现在，假设ProductID列是“高选择性”列（使用搜索条件中的任何ProductID值选择少于行总数的5％），使用索引咧查询应该更快速。

与在外键列（ProductID）上未创建索引的情况相比，确实是这样的，在这种情况下，进行全表扫描（扫描表中的所有相关页面以检索所需的数据）。但是，仍然有进一步的空间来改进这个查询

让我们假设Sales表包含10,000行，以下SQL选择400行（占总行数的4％）：

```sql
SELECT SalesDate, SalesPersonID FROM Sales WHERE ProductID = 112 
```

让我们试着理解这个SQL如何在SQL执行引擎中执行：

>1. Sales表在ProductID列上有一个非聚集索引,因此，它“寻找”非聚集索引树用于查找包含ProductID = 112的条目。
>2. 该索引页面还包含所有聚簇索引键（所有主键值，即SalesID，具有ProductID = 112，假定主键已在Sales表中创建）。
>3. 对于每个主键（这里为400），SQL Server引擎“查找”到聚集索引树中，以找到相应页面中的实际行位置。
>4. 对于每个主键，当找到时，SQL Server引擎从相应的行中选择SalesDate和SalesPersonID列值。

请注意在上面的步骤，在生产服务器中生成的执行计划可能和测试服务器中生成的不同，即使表和索引结构相同，这也表明在测试服务器中创建的索引可能会提高测试应用程序中的一些TSQL性能，但在生产数据库中创建相同的索引可能不会在生产应用程序中给您带来任何性能优势，为什么？因为测试环境中的SQL执行计划利用了新创建的索引，从而为您提供更好的性能，但是，由于某种原因，生产服务器中生成的执行计划可能根本不使用新创建的索引（例如，非聚簇索引列在生产服务器数据库中不是“高度”选择性的，这在测试服务器数据库中可能不同）

因此，在创建索引时，我们需要确保索引用来产生更快的结果。但是，我们怎么做呢？答案是，我们必须模拟生产服务器在测试服务器中的负载，然后需要创建适当的索引并测试这些，只有这样，如果新创建的索引提高了测试环境中的性能，这些将很可能提高生产环境中的性能

这应该很难，但幸运的是，我们有一些好的工具来做到这一点。

>1. 使用SQL Profiler在生产服务器中捕获跟踪,使用调整模板（我知道，建议不要在生产数据库中使用SQL Profiler，但有时必须在诊断生产中的性能问题时使用它）.如果您不熟悉此工具，或者需要使用SQL Profiler了解有关性能分析和跟踪的更多信息，请阅读[http://msdn.microsoft.com/en-us/library/ms181091.aspx](http://msdn.microsoft.com/en-us/library/ms181091.aspx)。
>2. 使用上一步中生成的跟踪文件在测试数据库服务器中使用Tuning Advisor 创建类似的加载，看看 Tuning Advisor 能给出一些什么建议，你可能会从 Tuning Advisor 获得一些很好的建议（因为Tuning Advisor使用从生产数据库生成的跟踪加载测试数据库，然后尝试生成最佳的索引建议），使用Tuning Advisor工具，您还可以创建它建议的索引，或者如果您需要了解有关使用Tuning Advisor的更多信息，请阅读[http://msdn.microsoft.com/en-us/library/ms166575.aspx](http://msdn.microsoft.com/en-us/library/ms166575.aspx)。

### 第三部：索引碎片整理

您在表中创建了所有适当的索引。或者索引已经存在于您的数据库表中，但是你可能得到良好性能.

有很大的机会发生索引碎片。

#### 什么是索引碎片

索引碎片是索引页由于对数据库中的表的大量插入，更新和删除操作而分裂的情况。如果索引具有高碎片，扫描/寻找索引需要很多时间，或者在执行查询时根本不使用索引（导致表扫描），因此，数据检索操作执行缓慢。

可能发生两种类型的碎片：

>* 内部碎片:发生的原因是索引页中的数据删除/更新操作，最终导致数据分布在数据页中索引稀疏（在页面中创建大量空行）。还导致索引/数据页的增加，增加查询执行时间。
>* 外部碎片:发生在索引/数据页面中的数据插入/更新操作，最终在文件系统中页面拆分和分配出现不连续的新索引/数据页面，这会降低确定查询结果的性能，数据库服务器不能利用预读操作，因为下一相关数据页不能保证是连续的，因为这些下一页可以在数据文件中的任何地方。

### 如何知道是否发生索引碎片

在数据库中执行以下SQL（以下SQL将在SQL Server 2005或更高版本的数据库中工作）。在以下查询中将数据库名称“AdventureWorks”替换为目标数据库名称

```sql
SELECT object_name(dt.object_id) Tablename,si.name
IndexName,dt.avg_fragmentation_in_percent AS
ExternalFragmentation,dt.avg_page_space_used_in_percent AS
InternalFragmentation
FROM
(
    SELECT object_id,index_id,avg_fragmentation_in_percent,avg_page_space_used_in_percent
    FROM sys.dm_db_index_physical_stats (db_id('AdventureWorks'),null,null,null,'DETAILED'
)
WHERE index_id <> 0) AS dt INNER JOIN sys.indexes si ON si.object_id=dt.object_id
AND si.index_id=dt.index_id AND dt.avg_fragmentation_in_percent>10
AND dt.avg_page_space_used_in_percent<75 ORDER BY avg_fragmentation_in_percent DESC 
```

上面的查询显示了'AdventureWorks'数据库的索引碎片信息如下：

<img src="https://www.codeproject.com/KB/database/OptimizeDBUseIndexing/IndexFragmentation.JPG" style="width:750px;"/>

分析结果，您可以使用以下规则确定索引碎片发生的位置：

>* ExternalFragmentation值> 10表示对应的索引发生了外部碎片
>* InternalFragmentation值<75表示发生相应索引的内部碎片

#### 如何碎片整理索引？

#### 你可以通过两种方式做到这一点

1. 重组碎片索引：执行以下命令来执行此操作：

```sql
ALTER INDEX ALL ON TableName REORGANIZE
```

2. 重建索引：执行以下命令：

```sql
ALTER INDEX ALL ON TableName REBUILD WITH (FILLFACTOR=90,ONLINE=ON) 
```

您还可以使用索引名称而不是上述查询中的“ALL”关键字重建或重组表中的各个索引,您还可以使用SQL Server Management Studio进行索引碎片整理。

<img src="https://www.codeproject.com/KB/database/OptimizeDBUseIndexing/SSMSDefragmentIndex.JPG" style="width:750px;"/>

#### 什么时候重组和什么时候重建索引？

当对应索引的“外部碎片”值在10-15之间且内部碎片值在60-75之间时，应该“重新组织”索引。否则，您应该重建索引

索引重建值得注意的是，在重建特定表的索引时，整个表将被锁定（这在索引重组的情况下不会发生），所以，对于生产数据库中的大表，这种锁定可能不是我们想要的，因为为该表重建索引可能需要几个小时才能完成，幸运的是，在SQL Server 2005中，有一个解决方案。在重建表的索引时，可以将ONLINE选项用作ON（参见上面给出的index rebuild命令），这将重建表的索引，并使表可用于事务。

### 最后的话

在数据库表中的所有合格列上创建一个索引真的很有用处，但是，如果你使用事务数据库（大多数时间进行更新操作的OLTP系统），可能不需要在每个合适的列上建立索引，事实上，在OLTP系统上创建重索引可能会降低整体数据库性能
