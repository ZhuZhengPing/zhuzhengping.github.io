### 第九步：组织数据库中的文件组和文件

创建SQL Server数据库时，数据库服务器在内部文件系统中创建多个文件。

>* .mdf 文件：这是主文件。每个数据库只能有一个主数据文件，所有系统对象保留在主数据文件中。
>* .ndf 文件：这是辅助文件。这是可选的。这些文件包含用户创建的对象。
>* .ldf 文件：这些是事务日志文件。这些文件可以是一个或多个。它们包含事务日志。

创建SQL Server数据库时，默认情况下将创建主数据文件和事务日志文件。当然，您可以修改这两个文件的默认属性。

<img src="https://www.codeproject.com/KB/database/OrganizeFilesAndPartition/Data_files.JPG" />

#### 文件组

数据库文件在逻辑上分组，以便在大型数据库上获得更好的性能和改进的管理.创建新的SQL Server数据库时，将创建主文件组，主数据文件包含在主文件组中，主组被标记为默认组，因此，每个新创建的用户对象都自动放置在主文件组

如果要在辅助数据文件中创建用户对象

>* 创建一个新文件组并将该文件组标记为默认的
>* 创建新的数据文件（.ndf文件），并将此数据文件的文件组设置为刚刚创建的新文件组

请注意，事务日志文件不包括在任何文件组中。

<img src="https://www.codeproject.com/KB/database/OrganizeFilesAndPartition/FileGroup.JPG" />

#### 文件/文件组组织最佳实践

当您有一个小或中等大小的数据库,那么在创建数据库时创建的默认文件/文件组可能就足够了.但是，当你的数据库有一个增长的趋势（比如，超过1000 MB）的大小，您可以（并且应该）在数据库中的文件/文件组组织中进行一些调整，以提高数据库性能。以下是您可以遵循的一些最佳做法

>* 主文件组必须完全分开，并且仅保留系统对象，并且不应在主文件组上创建用户定义的对象.主文件组不应设置为默认文件组.将系统对象与其他用户对象分离可以提高性能，在严重数据故障的情况下增强访问表的能力。
>* 如果系统中有N个物理磁盘驱动器可用，然后尝试为每个文件组创建N个文件，并将每个文件放在单独的磁盘中。这可以在多个磁盘上分配磁盘I / O负载，并且将提高性能。
>* 如果表访问频繁，并且包含索引，将表和索引放在单独的文件组中。能够更快地读取索引和表数据。
>* 如果表访问频繁，并且包含文本或图像列，创建单独的文件组，并将该文件组中的text，next和image列放在不同的物理磁盘上，并将这些表放在不同的文件组中。如果查询中不包含文本或图像，速度将会更快。
>* 将事务日志文件放在不同物理磁盘上,日志记录操作（事务日志写入操作）更加耗时，因此重要的是让磁盘上的日志具有良好的I / O性能。
>* 将“只读”表分配到标记为“只读”的文件组中,这将使得只读表中更快地进行数据检索。
>* 不要过多使用 “自动增长” 的功能，这会很消耗性能。 设置“自动增长”增量值，以便减少数据库大小增加的频率。

### 第十步：在大数据表中应用分区

表分区只是将大表分成多个较小的表，以便查询时扫描较少的数据量。

当你的数据库中有某个表数据库非常大时，并且你每次查询非常慢，你应该考虑以提高性能。

#### 水平分割

假设我们有一个包含1000万行的表，为了易于理解，让我们假设表具有自动增量主键字段（例如，ID）。我们可以将表的数据划分为10个单独的分区表，其中每个分区将包含1百万行，并且该分区将基于ID字段的值。也就是说，第一个分区将包含主键值在1-1000000范围内的那些行，而第二个分区将包含主键值在1000001-2000000范围内的行，以此类推。

#### 垂直分隔

假设我们有一个包含许多列和数百万行的表，表中的某些列在一些查询中经常使用，并且表中的大多数列不经常在查询中使用。

由于表的体积是巨大的（在列和行的数量），这个表的任何查询都会非常慢。因此，该表可以基于列的访问频率来分割。我们可以将表分成两个或多个表（分区），其中每个表将包含原始表中的几列。在我们的示例中，此表的分区应包含查询频繁的列，并且此表的另一个分区应包含其他查询不太频繁访问的列。

应用垂直分区的另一个好方法是将索引列和非索引列分割成单独的表，此外，垂直分区可以通过将LOB或VARCHARMAX列拆分为单独的表来完成。

与水平分区一样，垂直分区也可以提高查询性能，但是这种类型的分区要慎重进行，因为如果有任何查询涉及两个分区的列，则查询处理引擎将需要表的两个分区以查询数据，这又会降低性能。

在本文中，我们将仅关注水平分区

<b>分区最佳实践</b>

>* 考虑将大数据表分成不同的文件组，其中文件组中的每个文件都分散到单独的物理磁盘中。使数据库引擎能够更快地读取/写入数据操作。
>* 对于历史数据，根据时间考虑.例如，假设一个表有订单数据。要分区此表，请使用“订单日期”列来拆分表，以便创建包含每年销售数据的分区。

<b>怎么分区？</b>

假设，在我们的数据库中有一个Order表，其中包含4年（1999,2000,2001和2002）的订单数据，此表包含数百万行，我们想在这个表上应用分区。为此，需要执行以下任务：

<b>1. 将用户定义的文件组添加到数据库</b> 

使用以下SQL命令创建文件组：

```sql
ALTER DATABASE OrderDB ADD FILEGROUP [1999]

ALTER DATABASE OrderDB ADD FILE (NAME = N'1999', FILENAME
= N'C:\OrderDB\1999.ndf', SIZE = 5MB, MAXSIZE = 100MB, FILEGROWTH = 5MB) TO
FILEGROUP [1999]  
```

在这里，我们添加一个文件组'1999'，并添加一个辅助数据文件'C：\ OrderDB \ 1999.ndf'到这个文件组。因为我们想把我们的表分区放在不同的文件组中。

使用上面的SQL命令，创建三个其他文件组'2000'，'2001'和'2002'。这些文件组中的每一个都将在其相应的数据文件中存储一年的订单数据。


<b>2. 创建分区函数</b>

分区函数是定义用于分区数据的边界点的对象.以下命令创建分区函数：

```sql
CREATE PARTITION FUNCTION FNOrderDateRange (DateTime) AS
RANGE LEFT FOR VALUES ('19991231', '20001231', '20011231')
```

上述分区函数指定订单日期列的值介于以下之间：

```sql
DateTime <= 1999/12/31 would fall into 1<sup>st</sup> partition.
DateTime > 1999/12/31 and <= 2000/12/31 would fall info 2<sup>nd</sup> partition.
DateTime > 2000/12/31 and <= 2001/12/31 would fall info 3rd partition.
DateTime > 2001/12/31 would fall info 4th partition.
```

RANGE LEFT用于指定边界值应该落入左分区,例如，这里边界值1999/12/31落入第一分区中,如果我们指定RANGE RIGHT，那么边界值将落入右分区,在该示例中，边界值2000/12/31将落入第二分区中,并且小于此值的任何日期将落入第一个分区

<b>3. 创建分区模式</b>

将分区表/索引的分区映射到将用于存储分区的文件组。

以下命令创建分区模式：

```sql
CREATE PARTITION SCHEME OrderDatePScheme AS PARTITION FNOrderDateRange
TO ([1999], [2000], [2001], [2002])
```

在这里，我们指定：

```sql

The 2<sup>nd </sup>partition should go into the â€˜2000' file group
The 3<sup>rd</sup> partition should go into the â€˜2001' file group
The 4<sup>th</sup> partition should go into the â€˜2002' file group
```

<b>4. 在表上应用分区</b>

可以使用DROP INDEX语句删除聚簇索引，假设PK_Orders是表的主键，请使用以下命令删除主键，这将最终从表中删除聚集索引：

```sql
ALTER TABLE Orders DROP CONSTRAINT
PK_Orders
```

可以在分区方案上创建索引，如下所示：

```sql
CREATE UNIQUE CLUSTERED INDEX PK_Orders ON Orders(OrderDate) ON
OrderDatePScheme (OrderDate)
```

假设OrderDate列值在表中是唯一的,该表将根据指定的分区方案（OrderDatePScheme）进行分区,其内部使用分区功能将表分割成4个不同文件组中的4个较小部分。
