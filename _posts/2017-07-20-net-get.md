## 概念

每个程序都要使用资源，在面向对象的环境中，每个类型都代表可供程序使用的一种资源。要使用这些资源，必须为代表资源的类型分配内存。

>* 调用IL指令 newobj,为代表资源的类型分配内存（一般使用C# new 操作符来完成）。
>* 初始化内存，设置资源的初始化状态并使资源可用。类型的实例构造器负责设置初始状态
>* 访问类型的成员来使用资源(有必要可以重复)。
>* 摧毁资源的状态以进行清理
>* 释放内存。垃圾回收器独自负责这一步。









#### 从托管堆分配资源

对于托管堆，分配对象只需在指针上加上一个值，速度相当快。由于托管堆在内存中连续分配这些对象，所以会因为引用的“局部化”而获得性能上的提升。这意味着
进程的工作集会非常小，应用程序只需使用很少的内存，从而提高了速度。。， 开门， 额·  

但是内存不可能无限，所以CLR通过称为“垃圾回收”（GC）的技术删除堆中不需要的对象。


#### 垃圾回收算法

应用程序调用 new 操作符创建对象时，可能没有足够地址空间来分配该对象。发现空间不够，CLR就执行垃圾回收。

CLR开始GC时，首先暂停进程中的所有线程。然后,CLR进入GC的标记阶段。在这个阶段，CLR遍历堆中的所有对象，将同步块索引字段中的一位设为0。这表明所有对象都应删除。
然后，CLR检查所有活动根，查看它们引用了哪些对象。任何根引用了堆上的对象，CLR都会标记那个对象，也就是将该对象的同步块索引为设置为1.已标记的对象不能被
垃圾回收，CLR对堆中已标记的对象进行“乾坤大挪移”，压缩所有幸存下来的对象，使它们占用连续的内存空间。

#### 垃圾回收和调试

一旦根离开作用域，它引用的对象就会变成可回收，如下面的代码

```c#
public static class Program{
	public static void Main(){
		Timer t = new Timer(TimerCallback,null,0,2000);
		
		Console.ReadLine();
	}
	
	private static void TimerCallback(object o){
		Console.WriteLine("In TimerCallback: "+DateTime.Now);
		
		// 出于演出目的，强制执行一次垃圾回收
		GC.Collect();
	}
}
```

在命令行上，不用任何特殊编译器开关编译代码。运行可执行文件，会发现 TimerCallback 方法只被调用一次！

回收开始时，垃圾回收器检查应用程序的根，发现在初始化后，Main 方法再也没有用过变量t，就被回收了。

可试着像下面这样修改 Main 方法：

```c#
public static void Main(){
	public static void Main(){
		Timer t = new Timer(TimerCallback,null,0,2000);
		
		Console.ReadLine();
		
		// 在 ReadLine 之后引用 t （会被优化掉）
		t=null;
	}
}
```

执行上面的代码，TimerCallback 方法仍然只被调用一次。将局部变量或参数设置为null,等价于不引用该变量。

下面才是 Main 方法的正确修改方式：

```c#
public static void Main(){
	public static void Main(){
		Timer t = new Timer(TimerCallback,null,0,2000);
		
		Console.ReadLine();
		
		t.Dispose();
	}
}
```

#### 使用需要特殊清理的类型

你现在基本了解垃圾回收和托管堆的情况，大多数类型有内存就能正常工作。但有的类型除了内存还需要本机资源。例如 System.IO.FileStream 类型需要打开一个文件
并保存文件的句柄。然后，类型的 Read 和 Write 方法用句柄操作文件。

包含本机资源的类型被GC时，GC会回收对象在托管堆中使用的内存。但这样会造成本机资源的泄漏，这当然是不允许的。所以CLR提供了需要终结的机制，允许对象在被
判断为垃圾之后，但在对象内存被回收之前执行一些代码。CLR判定一个对象不可达时，对象将终结它自己，释放它包装的本机资源。

终极基类 System.Object 定义了受保护的虚方法 Finalize。垃圾回收器判定对象时垃圾后，会调用对象的 Finalize 方法。

```c#
internal sealed class SomeType{
	// 这是一个 Finalize 方法
	~SomeType(){
		// 这里的代码会进入 Finalize 方法
	}
}
```

被视为垃圾的对象在垃圾回收完毕后才调用 Finalize 方法，所以这些对象的内存不是马上被回收的，因为 Finalize 方法可能要执行访问字段的代码。可终结对象在回收时
必须存活，造成它被提升到另一代，使对象活的比正常时间长。这增大了内存耗用，所以应尽可能避免终结。

```c#
public abstract class SafeHandle:CriticalFinalizerObject,IDisposable{
	// 这是本机资源的句柄
	protected IntPtr handle;
	
	protected SafeHandle(IntPtr invalidHandleValue,Boolean ownsHandle){
		this.handle = invalidHandleValue;
		// 如果 ownsHandle 为 true, 那么这个从 SafeHandle 派生的对象被回收时，
		// 本机资源会被关闭
	}
	
	protected void SetHandle(IntPtr handle){
		this.handle = handle;
	}
	
	// 可调用 Dispose 显示释放资源
	// 这是 IDisposable 接口的 Dispose 方法
	public void Dispose(){ Dispose(true);}
	
	// 默认的 Dispose 实现（如下所示）正是我们希望的。强烈建议不要重写这个方法
	protected virtual void Dispose(Boolean disposing){
		// 这个默认实现会忽略 disposing 参数：
		// 如果资源已经释放，那么返回；
		// 如果 ownsHandle 为 false, 那么返回；
		// 设置一个标志来指明该资源已经释放；
		// 调用虚方法 ReleaseHandle；
		// 调用 GC.SuppressFinalize(this)方法来阻止调用 Finalize 方法；
		// 如果 ReleaseHandle 返回 true, 那么返回；
		// 如果走到这一步，就激活 releaseHandleFailed 托管调试助手（MDA）。
	}
	
	// 默认的 Finalize 实现（如下所示）正是我们希望的。强烈建议不要重写这个方法
	~SafeHandle(){ Dispose(false); }
	
	// 派生类要重写这个方法以实现释放资源的代码
	protected abstract Boolean ReleaseHandle();
	
	public void SetHandleAsInvalid(){
		// 设置标准来指出这个资源已经释放
		// 调用 GC.SuppressFinalize(this) 方法来阻止调用 Finalize 方法
	}
	
	public Boolean IsClosed{
		get {
			// 返回指出资源是否释放的一个标志
		}
	}
	
	public abstract Boolean IsInvalid{
		// 派生类要重写这个熟悉
		// 如果句柄的值不代表资源(通常意味着句柄为0或-1)，实现应返回true
		get;
	}
	
	// 以下三个方法涉及安全性和引用计数，本节最后会讨论它们
	public void DangerousAddRef(ref Boolean success) {...}
	public IntPtr DangerousGetHandle() {...}
	public void DangerousRelease() {...}
}
```

SafeHandle 类有两点需要注意。其一，它派生自 CriticalFinalizerObject ; 后者在 System.Runtime.ConstrainedExecution 命名空间定义。CLR 以特殊方式对待这个类
以及其派生类。CLR赋予这个类以下三个很酷的功能。

>* 首次构造任何 CriticalFinalizerObject 派生类型的对象时，CLR 立即对继承层次结构中所有 Finalize 方法进行 JIT 编译。内存紧张时，CLR可能找不到足够的内存来编译
Finalize 方法，这会阻止 Finalize 方法的执行，造成本机资源泄漏。另外，如果 Finalize 方法中的代码引用了另一个程序集中的类型，但CLR定位该程序集失败，那么资源将得不到释放。
>* CLR 是在调用了非 CriticalFinalizerObject 派生类型的 Finalize 方法之后，才调用 CriticalFinalizerObject 派生类型的 Finalize 方法。这样，托管资源类就可以
在它们的 Finalize 方法中成功地访问 CriticalFinalizerObject 派生类型的对象。
>* 如果 AppDomain 被一个宿主应用程序强行中断，CLR 将调用 CriticalFinalizerObject 派生类型的 Finalize 方法。

### 使用包装了本机资源的类型

假如要写代码来创建一个临时文件，向其中写入一些字节，然后删除文件。开始可能会像下面这样写代码

```c#
public static void Main(){
	Byte[] bytesToWrite = new Byte[]{1,2,3,4,5};
	FileStream fs = new FileStream("Temp.dat",FileMode.Create);
	fs.Write(bytesToWrite,0,bytesToWrite.Length);
	File.Delete("Temp.dat");
}
```

Delete 方法会抛出 System.IO.IOException 异常，并显示以下字符串消息：文件"Temp.dat"正由另一进程使用，因此该进程无法访问此文件。

幸好，FileStream 类实现了 IDisposable 接口。在实现中、会在 FileStream 对象的私有 SafeFileHandle 字段上调用 Dispose.现在就能修改代码来显示关闭文件，而不是
等着未来某个时候GC的发生。下面是修改后的源代码

```c#
public static void Main(){
	Byte[] bytesToWrite = new Byte[]{1,2,3,4,5};
	FileStream fs = new FileStream("Temp.dat",FileMode.Create);
	fs.Write(bytesToWrite,0,bytesToWrite.Length);
	fs.Dispose();
	File.Delete("Temp.dat");
}
```

现在，当调用File的Delete方法时，Windows发现该文件没有打开，所以能成功删除它。

注意，并非一定要调用 Dispose 才能保证本机资源得以清理。本机资源的清理最终总会发生，调用 Dispose 只是控制这个清理动作的发生时间。另外，调用 Dispose 不会将托管
对象从托管堆删除。只有在垃圾回收之后，托管堆中的内存才会得以回收。这意味着即使 dispose 了托管对象过去用过的任何本机资源，也能在托管对象上调用对象。

前面的例子展示了怎样显示调用类型的 Dispose 方法。如果决定显示调用 Dispose, 强烈建议将调用放到一个异常处理 finally 块中。

```c#
public static void Main(){
	Byte[] bytesToWrite = new Byte[]{1,2,3,4,5};
	FileStream fs = new FileStream("Temp.dat",FileMode.Create);
	try{
		fs.Write(bytesToWrite,0,bytesToWrite.Length);
	}finally{
		if(fs != null) fs.Dispose();
	}
	fs.Dispose();
	File.Delete("Temp.dat");
}
```

添加异常处理是正确的 ，而且应该坚持这样做。幸好，C#语言提供了一个 using 语句，它允许用简化代码的语法来获得和上述代码相同的结果。

```c#
public static void Main(){
	Byte[] bytesToWrite = new Byte[]{1,2,3,4,5};
	using(FileStream fs = new FileStream("Temp.dat",FileMode.Create)){
		fs.Write(bytesToWrite,0,bytesToWrite.Length);
	}
	File.Delete("Temp.dat");
}
```
